# CRM系统开发说明书

- 文档版本：V1.0
- 编写日期：2026-02-26
- 适用范围：基于现有Excel客户数据构建CRM数据库与客户/商机管理系统，并扩展智能分析与下一步建议能力。

## 1. 建设目标

本系统目标是将当前“静态表格管理”升级为“可持续更新、可分析、可建议下一步行动”的CRM系统，支持：

1. 客户资料统一管理（主数据中心化）。
2. 商机全流程管理（阶段推进、金额预测、风险预警）。
3. 客户互动与活动管理（拜访/电话/纪要/任务）。
4. 基于阶段Playbook的过程管理（销售与技术分工落地）。
5. 通过规则与AI实现动态分析和行动建议。

## 3. 技术选型建议（结合Mac环境）

### 3.1 数据库

1. MVP阶段（个人/小团队开发）：SQLite。
2. 多人协作与正式部署：MySQL（或PostgreSQL）。
3. 非强依赖微软生态时，不建议在Mac首选SQL Server。

### 3.2 后端与前端建议

1. 后端：Python FastAPI 或 Node.js NestJS。
2. 前端：React + Ant Design（或Vue + Element Plus）。
3. ETL：Python（pandas + SQLAlchemy）。
4. 报表：ECharts/Chart.js。

## 4. 系统总体架构

1. 数据采集层：Excel上传、模板识别、批次管理。
2. 数据处理层：字段映射、清洗、去重、增量合并（Upsert）。
3. 数据存储层：CRM业务库（客户、商机、活动、任务、阶段、建议）。
4. 业务服务层：客户360、商机管理、活动管理、看板分析、智能建议。
5. AI能力层：规则引擎 + 评分模型 + 文本生成（周报/建议/纪要）。

## 5. 数据库设计（核心表）

### 5.1 主实体

1. `accounts`（客户主表）
2. `opportunities`（商机主表）
3. `opportunity_line_items`（商机明细）
4. `activities`（活动记录）
5. `contacts`（联系人，后续补齐）

### 5.2 过程与规则实体

1. `stage_definitions`（阶段字典，Prospect~Closed Won/Lost）
2. `role_definitions`（角色字典：ISM、TO、TBM、RTM）
3. `stage_playbook_tasks`（每阶段标准动作模板）
4. `opportunity_stage_task_status`（商机动作完成状态）
5. `opportunity_stage_history`（阶段流转历史）
6. `opportunity_recommendations`（系统建议记录）

### 5.3 导入与审计实体

1. `import_batches`（导入批次）
2. `import_file_logs`（文件级日志）
3. `import_row_errors`（行级错误）
4. `raw_*`（原始落地表，保留追溯）

## 6. Excel导入便捷性设计（重点）

目标：保证“长期重复导入同格式Excel”时，操作最少、风险最低。

### 6.1 体验设计

1. 导入中心上传：拖拽多个文件，自动识别表类型。
2. 自动预检：展示将新增/更新/跳过/报错数量。
3. 一键执行：确认后批量入库。
4. 错误下载：导出失败记录与原因。
5. 批次回滚：支持按批次撤销。

### 6.2 技术规则

1. 固定模板识别：按 `文件名 + Sheet名` 自动映射。
2. 数据标准化：去除 `--`，统一日期与金额格式。
3. 幂等控制：`batch_id + row_hash` 防重复。
4. 增量合并：核心表执行Upsert。
5. 软删除策略：导入时不直接物理删库，保留历史。

### 6.3 主键与更新键建议

1. `accounts`：业务键优先 `客户名称（中）`（后续补唯一客户编码）。
2. `opportunities`：优先 `商业机会名称 + 客户名称`（如后续有机会ID则切换ID）。
3. `opportunity_line_items`：使用源 `ID` 作为外部明细ID。
4. `activities`：使用活动 `ID` 去重；若重复则按时间+内容合并。

## 7. 销售阶段与角色分工落地

已识别角色：

1. `TO`：Technical Owner（技术支持人员）
2. `TBM`：Technic(al) Business Manager（技术经理）
3. `RTM`：Regional Technical Manager（区域技术经理）
4. `ISM`：区域销售经理

落地原则：

1. 每个阶段必须绑定“销售动作 + 技术动作 + 交付物”。
2. 每个动作必须指定 `owner_role`（如ISM/TO）和 `due_rule`。
3. 阶段推进需满足最低门槛（动作完成率 + 客户信号）。

建议的阶段门禁（示例）：

1. Qualify -> Need：需完成关键人识别（UB/TB/EB）和痛点确认。
2. Need -> Upside：需提交As-Is/To-Be与初步配置建议。
3. Endorse -> Stretch：需完成方案确认与采购路径确认。
4. Commit -> Deliver：合同状态、实施计划、项目团队已就绪。

## 8. 智能化能力规划

### 8.1 第一层：动态分析（即刻可做）

1. 漏斗分析：阶段数量、金额、转化率、停留天数。
2. 预测分析：按阶段权重预测月/季签单。
3. 客户健康度：互动频次、活跃度、服务投入、商机热度。
4. 团队分析：按销售/技术负责人看推进效率与赢单率。
5. 风险榜单：高金额低活动、长期卡阶段、临近合同日未推进。

### 8.2 第二层：智能建议（规则+评分）

1. Next Best Action：给出下一步建议动作与责任人。
2. 商机优先级评分：金额、时效、阶段、动作完成度综合评分。
3. 阶段漂移预警：发现“阶段虚高”或“倒退风险”。
4. 客户沉睡预警：识别近N天无有效互动且Pipeline下降客户。

### 8.3 第三层：AI辅助内容生成

1. 自动生成商机周报/月报。
2. 自动生成拜访前简报（客户背景+关键问题+建议话术）。
3. 自动生成会议纪要摘要与待办清单。

## 9. 看板与指标口径

核心指标：

1. Pipeline总额、Open Pipeline金额、本月预计签约额。
2. 阶段转化率（Prospect->Qualify->Need->...）。
3. 阶段平均停留天数与超时率。
4. 赢单率（按行业/区域/负责人/客户分级）。
5. 活动覆盖率（近30天有有效活动的商机占比）。
6. Playbook完成率（阶段关键动作完成率）。

## 10. 分期实施计划

### Phase 1（2-3周）

1. 建库与基础导入（核心4表）。
2. 客户360与商机列表/详情。
3. 基础漏斗与金额看板。

### Phase 2（3-4周）

1. 导入中心（预检、错误报告、批次回滚）。
2. 阶段Playbook配置与执行跟踪。
3. 风险预警与规则化建议。

### Phase 3（4周+）

1. 赢单概率模型（可解释评分）。
2. AI文本生成（建议、周报、纪要）。
3. 预测优化与策略复盘闭环。

## 11. 验收标准

1. 核心表导入成功率 >= 99%。
2. 重复导入不产生重复业务数据（幂等）。
3. 看板与原始汇总表口径差异 <= 1%。
4. 商机阶段动作完成记录可追溯。
5. 系统可输出可执行“下一步建议”。
